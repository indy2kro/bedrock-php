# Bedrock PHP

## Core Values (Operational)

### Purpose of the Core Values

The core values of Bedrock PHP exist to translate philosophy into **operational instinct**.

They are not cultural slogans or aspirational statements.
They are decision-making shortcuts intended to guide choices **when the rules alone are insufficient**.

Each value exists to reduce hesitation, not to encourage debate.

---

### Value 1: Stability Over Novelty

Bedrock PHP prioritizes stability over novelty.

This means:

- choosing tools with known failure modes
- preferring older, well-understood solutions
- resisting upgrades without concrete operational benefit

Novelty is treated as a cost.

Security and bugfix maintenance is mandatory and is not considered novelty.

If a newer option does not clearly reduce operational risk, it is rejected.

Bedrock PHP assumes that:

- most novelty benefits are overstated
- most novelty costs are discovered late

---

### Value 2: Predictability Over Flexibility

Bedrock PHP values predictability more than flexibility.

Flexible systems:
- invite variation
- accumulate edge cases
- shift decisions to runtime

Predictable systems:
- constrain behavior
- surface problems early
- are easier to reason about under pressure

When forced to choose, Bedrock PHP limits options even at the cost of adaptability.

---

### Value 3: Explicit Limits Over Implicit Growth

Bedrock PHP requires explicit limits.

Limits define:
- when to stop
- when to do nothing
- when not to optimize

Implicit growth paths are treated as liabilities.

If a system can grow without encountering a hard boundary, that boundary is missing.

---

### Value 4: Deletion Over Abstraction

Bedrock PHP prefers deletion to abstraction.

Abstractions:
- hide complexity
- delay understanding
- are difficult to remove

Deletion:
- reduces surface area
- clarifies intent
- lowers long-term cost

When complexity arises, the first response is to remove, not to generalize.

---

### Value 5: Operational Clarity Over Theoretical Correctness

Bedrock PHP optimizes for clarity during incidents, not theoretical elegance.

A solution is considered correct if:

- it can be understood quickly
- it behaves consistently
- it fails in obvious ways

Theoretical improvements that increase cognitive load are rejected.

---

### Value 6: Defaults That End Discussions

Bedrock PHP defaults exist to **end discussion**, not to invite refinement.

Defaults are chosen to be:

- good enough for most cases
- difficult to misuse
- resistant to over-optimization

If a default sparks ongoing debate, it has failed its purpose.

---

### Value 7: Constraints Over Coordination

Bedrock PHP favors constraints over coordination.

Coordination:
- requires meetings
- depends on shared context
- degrades as teams grow or change

Constraints:
- apply consistently
- require no discussion
- survive personnel changes

When possible, Bedrock PHP encodes decisions as constraints rather than processes.

---

### Value 8: Known Failure Modes Over Hidden Risk

Bedrock PHP prefers systems that fail visibly and predictably.

Hidden risk is more dangerous than known limitation.

A system that fails loudly at known thresholds is preferred to one that degrades silently.

---

### Value 9: Boring by Design

Bedrock PHP is intentionally boring.

Boring systems:
- are easier to staff
- are easier to maintain
- age more gracefully

Excitement is not treated as a success metric.

If a design decision feels exciting, it deserves extra scrutiny.

---

### Value 10: Longevity Over Local Optimization

Bedrock PHP optimizes for systems that remain understandable years later.

Short-term gains that:
- lock in complexity
- obscure intent
- depend on fragile knowledge

are rejected, even if they improve immediate metrics.

---

### Closing Statement

These values are not independent.

Together, they bias Bedrock PHP toward:

- fewer decisions
- fewer surprises
- fewer irreversible commitments

When individual rules are unclear or incomplete, these values define the intended direction.

They exist to help teams choose the *least dangerous* path, not the most impressive one.
